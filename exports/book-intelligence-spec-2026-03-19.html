Hard Constraints

These rules are absolute. Every agent referencing this document must treat these as inviolable.

  1. DATA BOUNDARY: All book intelligence is derived from EXTERNAL PUBLIC SOURCES (ISBNDB, Open Library, web scraping, AI analysis of publicly available information about books). Bookmarked does NOT read, scan, or process actual book text, PDFs, ePubs, or any digital version of a book's content. The Content Scanner (Layer 3) is a SEPARATE future capability requiring publisher permission. It is OUT OF SCOPE for this specification.

  2. LANGUAGE BOUNDARY: Never generate UI copy, prompts, descriptions, or documentation that implies Bookmarked has read the book. Use: "external sources report" / "publicly available information indicates" / "based on metadata and public reviews." Do NOT use: "the book contains" / "what's inside this book" / "what's in this book."

  3. NOT A RATING SYSTEM: Bookmarked does not rate books. We surface intelligence from multiple sources so humans can make informed decisions. Framing our output as "ratings" risks being perceived as censorship. Do NOT use: "Bookmarked rates" / "our rating" / "we rate" / "Bookmarked's rating" / "rated by Bookmarked." Use: "AI-generated assessment" / "based on external sources" / "intelligence from public data."

  4. NOT A REPLACEMENT FOR LIBRARIANS: Librarians make judgment calls. Book Intelligence provides the data infrastructure they never had. Never position the system as replacing professional judgment. Do NOT use: "replaces librarians" / "instead of a librarian" / "no librarian needed" / "eliminates the need for" / "automates the librarian's job." Use: "supports" / "empowers" / "provides data for" / "surfaces intelligence so librarians can."

  5. NOT A CENSOR: We do not decide what books are appropriate. We provide data for districts to make defensible decisions within their regulatory framework.

  6. ANTI-GAMING: Community actions (verifications, disputes, source reports, accuracy feedback) are transparent, attributed, and additive — never subtractive. Community input adds a layer alongside AI-generated data; it never overwrites or removes the baseline. Administrators always see the same data as librarians. Cross-district patterns expose outliers. The system must account for practitioners who may attempt to suppress intelligence they philosophically disagree with — transparency and attribution are the primary defenses.

Trust Hierarchy & Confidence

This section defines the credibility model for all Book Intelligence data. Every field in BI carries a trust tier and a confidence indicator. These concepts are referenced throughout Parts 10-14.

Trust Tiers

Tier Name Definition Criteria Visual Indicator
1 AI-Generated Data produced by the BI pipeline from public sources Default tier for all pipeline output No special badge — baseline
2 Community-Informed Data that has received practitioner feedback (verifications, disputes, accuracy votes) At least 1 community action on the field "Community-Informed" badge with count
3 Bookmarked Verified Data confirmed or contradicted by ground truth from connected library systems District library system data directly confirms or contradicts the intelligence "Bookmarked Verified" badge with verification date

Freshness Dimension

Trust signals include a verification date. When significant new intelligence arrives after verification (new flags, source changes, rating updates), the verification status shifts to: "Verified — new intelligence available since [date]." This nudges re-review without invalidating the original verification.

Confidence Indicators

BI assesses its own certainty per field based on available evidence:

Indicator Meaning Criteria
Strong signal Multiple sources and/or community verification support this value 3+ public sources OR 3+ community verifications
Moderate signal Some evidence supports this value 1-2 public sources, no community feedback
Limited data Minimal evidence available — apply professional judgment 0-1 public sources, AI inference with thin data

Confidence indicators help the Prioritization Engine: books with low-confidence, high-stakes ratings (Tier 1 fields with limited data) get escalated for human review.

Part 1: What Book Intelligence Is

Specification

Book Intelligence is Bookmarked's system for aggregating publicly available information about books — what external sources report about a book, what's happening around books in the regulatory and cultural landscape, and how districts across the network are handling specific titles.

It is not a module or a tab. It is the intelligence engine underneath every surface in OnShelf.

Field Group C: Content Ratings

Three content dimensions rated on a severity scale. These directly inform compliance decisions — a "Moderate" or "Explicit" sexual content rating has regulatory implications under SB 13 and SB 412.

Regulatory context: Under SB 412, school personnel face criminal liability for "sexually explicit" material in K-12 settings. The accuracy bar for Sexual Content ratings is therefore higher than for other fields. A false "None" on a book with explicit content creates legal exposure for the district.

Field Type Source Source Boundary Tier Status Known Issues
Violence Enum (None/Mild/Moderate/Explicit) + descriptive text AI-generated AI analysis of public reviews and metadata — NOT from reading the book 2 LIVE Unaudited. AI may over- or under-rate.
Sexual Content Enum (None/Mild/Implied/Moderate/Explicit) + descriptive text AI-generated AI analysis of public reviews and metadata — NOT from reading the book 1 LIVE Scale inconsistency: "Implied" appears in UI but isn't in the standard 4-point scale. Highest regulatory weight.
Language Enum (None/Mild/Moderate/Explicit) + descriptive text AI-generated AI analysis of public reviews and metadata — NOT from reading the book 2 LIVE Verify: does every book get descriptive text, or only some?

Scale definitions:

  • None: Not present.
  • Mild: Present but not graphic (violence: verbal confrontations; sexual: romance/kissing; language: occasional damn/hell).
  • Implied: (Sexual Content only) Sexual activity suggested but not described. Non-standard value — needs standardization.
  • Moderate: Present with some detail.
  • Explicit: Graphic, detailed descriptions.

Specification

The Book Intelligence pipeline runs as a 5-stage processing pipeline. Documented in Confluence (Bookmarked Intelligence Core, page 43188225).

Data Flow:

District uploads ISBNs (CSV or manual entry)
        ↓
ISBN Validation & Deduplication
        ↓
External Data Lookup (ISBNDB, Open Library)
  → Identity & Metadata fields populated
        ↓
AI Enrichment Pipeline
  → Content ratings, themes, educational value,
     grade suitability, risk level, AI summary
  → All derived from PUBLIC METADATA — not from reading the book
        ↓
Flag Detection & Classification
  → Ban/challenge data, organization flags,
     news mentions, school district actions
  → All from PUBLIC WEB SOURCES
        ↓
Data stored in database
        ↓
Intelligence surfaces in UI

Part 4: Accuracy Audit Framework

Specification

Book Intelligence has never been systematically audited for accuracy. This section defines what "correct" looks like and how to validate it.

Accuracy Priority Tiers

Tier Fields Standard Why
1: Compliance-Critical Sexual Content, Ban Status, Challenge Risk, Grade Level (Elementary) Must be accurate. False negatives unacceptable. SB 412 criminal liability. Districts rely on these for defensible decisions.
2: Decision-Informing Violence, Language, Parental Guidance, Content Warnings, Potential Concerns, Reading Level Should be accurate. Errors are problematic but not legally dangerous. Informs librarian/administrator decisions. Wrong data → wrong decisions.

District-Level Ground Truth

Status: PLANNED — This is Bookmarked's most defensible competitive advantage.

Bookmarked connects to actual library systems. This means when an external source claims a book has been banned or challenged in a district, and that district is a Bookmarked customer, Bookmarked can verify the claim against ground truth.

How it works:

  1. PEN America (or any external source) claims "Book X is banned in District Y"
  2. District Y is a Bookmarked customer
  3. Bookmarked connects to District Y's library system
  4. Book X is present on the shelf in 14 of 19 District Y libraries
  5. District Y's librarian has marked it "Reviewed — Retained"

This works in both directions:

  • Book IS on shelves despite ban claims → contradicts the claim
  • Book is NOT on shelves and district confirms removal → confirms the claim

At 150 districts, Bookmarked already has meaningful coverage. At 500, this becomes the definitive national authority on what's actually happening vs. what's being claimed. No other organization has this data because no other organization connects to actual library systems.

Bookmarked becomes the source of truth for actual bans and challenges. Every other organization reports claims. Bookmarked reports verifiable facts.

Part 10: Trust, Confidence & Anti-Gaming

Status: DESIGNED

Specification

This section defines how BI manages the trustworthiness of its own data — the architectural constraints that ensure community participation improves accuracy without enabling manipulation.

Trust Tier Implementation

Every field on the Intelligence Card displays its current trust tier (see Trust Hierarchy section in front matter). The trust tier advances as community actions accumulate:

Transition Trigger Effect
AI-Generated → Community-Informed First verification, dispute, or accuracy vote on the field Badge appears: "Community-Informed (1 action)"
Community-Informed → Bookmarked Verified District library system data confirms or contradicts the field Badge upgrades: "Bookmarked Verified [date]"
Any tier → "New intelligence available" Significant new data arrives post-verification Freshness nudge appears without invalidating prior verification

Confidence Model

BI assesses its own certainty per field and displays it to users. See Trust Hierarchy section for the three confidence levels (Strong signal, Moderate signal, Limited data).

How confidence is calculated:

Factor Weight Example
Number of public sources High A book with 50 reviews produces better AI ratings than one with 2
Number of community actions High 12 verifications > 1 verification
Source diversity Medium Flags from 8 organizations > 8 flags from 1 organization
Flag consistency Medium All sources agree > sources contradict each other
Data recency Medium Recent data weighted more than stale data

Anti-Gaming Architecture

The problem: Community features create an attack surface. A librarian who disagrees with a book's presence could:

  • Verify an inaccurate "None" rating to keep a book she likes
  • Dispute an accurate "Explicit" rating to prevent review
  • Submit misleading accuracy feedback
  • Report credible sources as "unreliable" to suppress flags
  • Use collaborative notes to build a case for a predetermined outcome

The defense — system-wide principles (reference: Hard Constraint #6):

  1. Transparent: Every community action is tied to a named user at a named district. No anonymous influence.
  2. Attributed: Actions are visible to the user's administrators. A librarian cannot suppress information from her superintendent.
  3. Additive: Community input adds a layer — it never overwrites or removes the AI-generated baseline. Both the AI value and the community signal are always visible.
  4. Cross-district exposure: If one librarian is disputing ratings that 12 other districts verified, that pattern is visible. Outlier behavior becomes detectable.
  5. Threshold-based influence: A single action doesn't move the needle. Multiple independent actions from different districts carry weight. One vote doesn't change a confidence indicator.

Specifically for source reporting (see Part 12, Phase 2):

  • Reports don't remove sources — they add a credibility signal
  • Structured reasons required (dropdown, not freeform)
  • Multiple independent reports from different districts required for downweighting

Part 14: The Learning System

Status: VISION — Strategic concepts for how BI compounds over time. Not yet scoped for implementation. Included for directional planning.

Specification

Everything in Parts 10-13 produces data. This section describes how that data makes the system better tomorrow than it is today. The Learning System is what turns BI from a static tool into a compounding asset.

AI Model Feedback Loop (VISION)

Every community verification, dispute, and accuracy vote is a signal about whether the AI got it right. Today, the AI enrichment runs once at import and never improves.

The vision: Accumulate thousands of human judgments over time. When the AI is wrong in predictable patterns (consistently over-rates violence in historical fiction, consistently under-rates sexual content in YA), adjust the prompts, add rules, or eventually fine-tune.

Not fine-tuning on day one. Start with: (1) identify systematic patterns in community feedback, (2) manually adjust enrichment prompts to address patterns, (3) measure whether adjustments improve accuracy, (4) iterate.

Source Quality Evolution (VISION)

Connects to: Source Intelligence System (Part 11).

Today, all flag sources are treated equally. With source unreliable reporting (Part 12) plus District-Level Ground Truth (Part 6), the system learns which sources are accurate over time.

The vision: If Source X has been contradicted by Bookmarked Verified data 40% of the time, that source gets downweighted automatically. If Source Y has been confirmed 95% of the time, it gets elevated. No human has to maintain a source quality list — the system learns it from usage.

Predictive Intelligence (VISION)

Once BI has enough data, it can start predicting: "Books with these characteristics (subject matter, author history, publisher, themes) have a 73% chance of being flagged within 12 months."

Not speculative — pattern recognition across the dataset. A librarian adding a book sees: "No current flags. Similar books in the Bookmarked network were flagged within 6 months 60% of the time."

Open question (Q12): At 150 districts, is there enough signal for meaningful predictions? What's the minimum dataset? This needs analysis before scoping.

Network Effect Compounding (VISION)

Each new district makes the system measurably better for all existing districts:

More districts = Better ...
More verifications Higher confidence scores across the network
More collection data Better peer benchmarking granularity
More flag reports Faster source quality learning
More review outcomes Richer "case law" for decision benchmarking
More ground truth data Stronger verification authority

At 150 districts: Useful dataset. Meaningful signals. At 500 districts: Patterns emerge that no individual district could see. At 2,000 districts: Real-time visibility into every significant trend in K-12 libraries.

Accuracy Metrics Dashboard — Internal (VISION)

A dashboard for Bookmarked (not customers) showing BI system health:

Metric Purpose
Accuracy rate per field Which fields are most/least accurate?
Accuracy by trust tier Is Community-Informed data actually better than AI-Generated?
Most-disputed AI assessments Where is the AI systematically wrong?
Source reliability rankings Which sources are most/least accurate?
Prediction accuracy over time Are predictions improving?

Value: Without this, the product team is flying blind on quality. With it, engineering effort can be directed to the highest-impact improvements.

Curriculum and Legislative Sync (VISION)

Laws change. Curriculum standards update. The books that matter shift.

The vision: When a new state passes a book regulation law, every book in the system gets re-assessed against the new regulatory context — not manually, but as an automated re-enrichment pass. Same for curriculum: when Texas updates TEKS, books tagged as curriculum-aligned get re-validated.

Engineering question (Q11): How would batch re-enrichment work? Cost? Performance? Timeline?
